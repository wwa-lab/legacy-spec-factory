Vendor BRD AI POC——SME Review 报告

1. Review 结论

评审结论：有条件接受，但尚不具备正式 BRD 审批条件。

该 POC 展示了一个面向 RPG/COBOL 类遗留批处理应用的反向工程分析工作台，能够将源代码分析结果组织为候选业务对象、持久化实体、接口、任务、规则、分支、流程视图、需求澄清问题，以及 BRS/FRS 类文档。

但是，演示尚未证明生成内容可以作为已批准的业务需求基线。目前输出至少混合了以下四类内容：

  1. 遗留系统技术分析；
  2. AI 推断的业务概念和规则；
  3. SME 需求澄清材料；
  4. BRD/FRS、角色矩阵、接口和验收测试类内容。

因此存在一个实质性风险：格式完整、视觉专业的文档可能被误认为业务事实，而其中部分内容实际上只是 AI 推断或代码抽取出的候选项。

建议处置：可以作为内部分析 POC 继续推进，并进入受控 Pilot；但当前输出不应直接作为正式 BRD、迁移基线、正式规格、目标系统设计、审计基线或 SDD handoff 使用。

1.1 管理层初步 Review 补充

管理层初步评审认为，Vendor 展示的自动建模、BPMN 流程预览、需求文档生成、自动需求澄清和语义对话具有较好的产品价值；但按大模块描述提取基础需求并非明显差异化能力，Legacy Spec Factory 也具备结合模块描述、文档和代码生成模块级候选需求的基础。

需要明确的是，仅基于模块描述生成的内容只能处于 candidate 或 poc_draft 状态；在没有源码证据、运行时证据或具名 SME 决定前，不得升级为 approved BRD。

管理层同时指出两个尚未被演示证明、但对 IBM i 遗留系统分析具有决定性影响的问题：

  1. Vendor 如何完整处理超过 10K 行的大型程序；
  2. Vendor 如何基于真实调用关系，从跨程序 Program Chain 中提取端到端业务需求。

因此，这两项应成为受控 Pilot 的强制 Challenge Case 和 Pilot Exit Gate，而不是普通的补充问题。Vendor 即使在一般功能评分上达标，只要任一强制用例不通过，也不得将其输出用于正式 BRD。

2. 评审阶段与严重度口径

本报告按三个不同 Gate 判断 Vendor POC。某项发现被标为“阻断”，必须同时说明它阻断的是哪个 Gate；不得把“阻断正式 BRD 审批”误读为“禁止使用脱敏数据开展受控 POC”。

Gate目的当前结论
Sandbox POC使用模拟、合成或经批准脱敏的材料验证建模、检索和交互能力有条件进入

Formal BRD Approval允许生成结果成为正式业务需求或迁移基线不具备条件

4. POC 的优点

4.1 分析到评审的工作流方向正确

从程序/源码分析，到任务、数据、规则、流程、问题和文档视图的链路方向合理。SME 可以检查具体候选项，而不必直接面对一份没有结构的长篇 AI 文本。

4.2 已经体现出证据意识

界面展示了源代码行号、程序/对象标识、证据接口数量以及来源/证据列。生成文档也尝试携带来源引用和证据等级。这是构建可审计反向工程流程的正确基础。

4.3 包含人工需求澄清环节

自动需求澄清是有价值的功能。POC 已经认识到：代码不能完全说明业务意图，必须将问题返回给业务或技术 SME 确认。

4.4 草稿输出具备较强实用性

POC 输出包括业务章节、流程步骤、角色、接口、数据对象和验收测试类行项目。如果进一步增加清晰的状态标签和证据链接，这些内容可以加速发现和评审工作。

7. 未加控制直接采用的主要风险

  1. 业务确定性虚高：代码模式被直接转换为需求，没有区分实现行为和业务意图。
  2. 审批责任被稀释：格式完整的文档使推断内容看起来像 SME 已批准的事实。
  3. 批处理行为不完整：重启、重跑、恢复、控制总数、调度窗口和运维异常容易被静态模型遗漏。
  4. 可追溯性断裂：来源名称或摘要不足以重现某条陈述背后的确切证据。
  5. 下游污染：草稿 BRD 可能在正式审批前被复用为 FRS、验收标准、目标设计或迁移范围。
  6. 敏感数据暴露：代码和生产样本可能在没有批准的数据处理设计下跨越租户或模型边界。

8. 受控 Pilot 的最低验收标准

Vendor 应在一个范围明确的能力上，使用已标注样本和指定 SME，证明以下内容：

  • 提供不可变的源码快照 ID、授权状态、敏感性、脱敏状态和精确坐标的证据清单。
  • 至少抽取 20 条行为/规则/流程进行抽样验证，SME 可以在一到两步内从文档陈述导航到源证据。
  • 分别呈现 observed behavior、inferred rule、candidate object、open question 和 SME-approved decision。
  • 问题流程包含负责人、SME 角色、回答、证据、日期、决策状态和审计历史。
  • 矛盾和缺失证据必须阻断审批，不能静默选择某一个答案。
  • 使用相同源码快照、模型/Prompt 版本和配置重复运行，并在发生变化时生成机器可读差异。
  • 明确批处理控制：触发、调度、重启/恢复、重试、重复处理、部分失败、控制总数、对账和人工干预。
  • 生成包含功能目的、场景、渠道/触点、接口、流程、校验、错误处理和依赖关系的 BRD Package，并显示每一节的证据覆盖率和未解决 TBD。
  • 生成正式签署记录，包含 SME 姓名、角色、日期、范围和限制；任何 AI 生成内容都不得在没有该记录的情况下升级为 approved。
  • 提供源代码、日志、数据抽取、Prompt、模型托管、保留期限和访问控制方面的安全/数据处理证据。

8.1 强制 Challenge Case 与 Pilot Exit Gate

Challenge Case A：10K+ 行大型程序

选择一个 10K-20K 行的真实 RPG 或 COBOL 程序，并验证：

  • 不允许因上下文窗口、解析器限制或超时发生静默截断；
  • 提供覆盖整个源码范围的读取清单、分段边界、Snapshot Hash 和处理状态；
  • 列出全部 Procedure、Subroutine 和主要分支，无法分析的区域必须显式标记；
  • 保持跨分析分段的变量、状态、文件访问和控制流关系；
  • 关键正常路径、异常路径和业务规则可以回到稳定源码坐标；
  • 在相同 Snapshot、模型、Prompt 和配置下重跑，核心结果一致或差异可解释。

Challenge Case B：Program Chain

选择一条包含 5-10 个程序的真实业务链，至少覆盖 CL 与 RPG/COBOL、数据库文件、一个外部接口和一个异常/重启路径，并验证：

  • SME 提供的 Program 导航顺序只能作为分析范围，不能直接当作真实调用链；
  • 每条被声明为 confirmed 的调用边必须有源码或运行时证据；
  • 无法确认的动态调用必须标记为 unresolved，不得补写为确定关系；
  • 输出跨程序字段血缘、状态码和参数传递、数据库持久化、异常传播及最终业务结果；
  • 覆盖 Commit/Rollback、重试、重启、部分失败、控制总数和人工干预；
  • BRD 中每个关键端到端步骤均可回到对应程序、调用边和证据坐标。

上述两项是 Pilot Exit 的强制通过项，不参与加权平均抵消：任何一项失败，都阻断正式 BRD Approval。对真实生产数据开展测试前，仍须先通过安全、授权和数据处理 Gate。

8.4 Abort Criteria

出现以下任一情况时，应暂停或终止 Pilot，而不是用更多生成页面掩盖质量问题：

  • 未经批准的生产数据、客户代码或敏感日志出域；
  • Vendor 无法说明数据保留、模型训练使用、租户隔离或删除机制；
  • 抽样中的 unsupported claim 超过 5%，或关键业务陈述无法回到稳定证据；
  • observed behavior 的 SME 修正率持续高于 20%；
  • 相同源码、模型、Prompt 和配置重跑后的核心结果无法解释地变化；
  • 系统隐藏矛盾证据、自动选择答案或把 unresolved 内容标为 confirmed；
  • Vendor 无法导出结构化证据、状态和 SME 决策记录；
  • 10K+ 行大型程序出现静默截断、未登记的源码区间或跨段关键语义丢失；
  • Program Chain 将 SME 导航顺序误作调用链、生成无证据调用边，或遗漏关键异常/重启结果；
  • 两轮评审后，SME Review 时间相对基线未减少至少 30%，且质量没有可量化提升。

5%、20% 和 30% 是建议起始阈值，Pilot kickoff 时可根据样本复杂度调整，但安全、证据伪造和审批越权属于零容忍项。

9. 建议的 Pilot 范围和评审方法

建议选择一个范围明确的批处理能力，包含：

  • 5-10 个程序，并至少包含一个 10K+ 行大型程序；
  • 1 条正常路径和至少 3 条异常路径；
  • 至少 1 个外部接口和 1 个持久化数据结构；
  • 可取得的源代码行号证据和少量运行时样本；
  • 1 名业务 SME、1 名 IBM i 技术 SME 和 1 名运维/安全评审人。

评审应采用分层抽样，而不是只看最漂亮的页面：

  1. 5 条高置信度 observed behavior；
  2. 5 条 inferred rule；
  3. 5 个候选数据/业务对象；
  4. 5 条流程或异常路径；
  5. 全部未解决和矛盾项。

Pilot 应测量准确率、证据链接成功率、SME 修正率、问题关闭率、重复运行一致性和节省的评审时间。完成标准应基于这些指标，而不是生成页数或抽取对象数量。

9.1 Pilot 资源与时间盒

建议以一个业务能力为单位，安排 1-2 周受控 Pilot：

  • 5-10 个程序，1 条正常路径和至少 3 条异常路径；
  • 20-30 条分层抽样声明；
  • 1 名业务 SME、1 名 IBM i 技术 SME、1 名运维/安全评审人；
  • Vendor 和 Legacy Spec Factory 各指定 1 名技术集成负责人。

工作量不应简单按程序数量线性估算；异常路径、外部接口、运行时证据和文档质量通常比程序数量更能决定评审成本。

10. 最终 SME 建议

建议：有条件推进。

该 POC 值得继续，因为它确实针对遗留系统发现阶段的一个实际瓶颈：将难以理解的遗留源代码分析转化为可视化、可评审的业务材料。下一阶段应优先强化证据和治理层，再在一个受控能力上验证语义准确性。

从产品能力看，自动建模、BPMN、需求文档生成、自动需求澄清和语义对话值得继续评估；从技术适用性看，最终判断必须取决于 10K+ 行大型程序和 Program Chain 两项 Challenge Case，而不能依据 UI 完整度、生成页数或对象数量。
