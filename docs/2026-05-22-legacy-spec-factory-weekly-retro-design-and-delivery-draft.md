这篇可以看作 `legacy-spec-factory` 工程系列的第二篇。第一篇是 Dogo 写的《千万行老代码里的需求，怎么反推出来：7×24 软件工厂的需求治理（上）》，重点讲的是老系统需求反推的方法论基础：为什么不能只靠访谈和旧文档，为什么必须从代码、数据、流程和 SME 判断里反推真实需求。

前一篇里，Dogo 讲的是一个很关键的问题：

**千万行老代码里的需求，不是采集出来的，而是反推出来的。**

这句话我非常认同。

因为在大型卡类核心、支付核心、银行后台这类老系统里，业务口头描述的“应该如此”，和系统真实运行的“实际上如此”，很多时候不是一回事。

方法论上，必须先承认这一点。

所以前一篇讲“四层逆流转”、讲“源码切片”、讲“数据字典”、讲“螺旋迭代”、讲“SME 终审”，本质上是在回答一个认知层面的问题：

> **老系统需求治理，到底应该从哪里出发。**

但这一个星期，我们围绕 `legacy-spec-factory` 反复讨论的，已经不是“方向对不对”了。

我们在推进的，是下一步：

> **如果这套方法论成立，怎么把它设计成一条真正能跑起来的工程链路。**

也就是说，我们不再满足于“知道该怎么想”。

我们开始认真回答“团队该怎么做”。

包括：

- 哪些输入该先准备好
- 哪些分析可以交给 AI
- 哪些判断必须交给 SME
- RAG 在这里到底扮演什么角色
- 什么内容能进入正式 spec
- 什么内容只能停留在推断和待确认
- 最后怎么把这一切交给下游 Java/cloud SDLC，而不是停留在一堆好看的分析文档里

这一周的探索，本质上是在做这件事。

如果要用一句话概括，就是：

**前一篇解决的是“反推需求的方法论”；这一周我们做的，是把这套方法论压成一条证据驱动、可评审、可交付、可继续被 AI 消费的生产线。**

## 一、第一件事，不是补功能，而是先把边界切清楚

这一周最重要的收敛，不是多了几个 skill，也不是多了几份文档。

而是我们先把边界想明白了。

过去一谈遗留系统治理，最容易犯的错，就是想做一个“大而全”的东西：

既想做代码理解，
又想做数据字典，
又想做知识库，
又想做需求梳理，
又想做测试设计，
最后还想直接生成新系统设计。

结果是每件事都沾一点，但没有一件事真正站稳。

所以这一周，我们做的第一件事，是把 `legacy-spec-factory` 的职责重新说清楚：

**它不是全字典中心。**

全域标准字段、字段 owner、标准口径、客户视图和内部视图的分类边界，这些应该由企业级的“全字典中心”来承接。

那是更上层的标准语义治理平台。

`legacy-spec-factory` 不应该重复造一套。

它真正负责的，是另一件更贴近 modernization 落地的问题：

**消费上游已经确认过的标准字段和业务语义，再把 legacy 代码、DDS、运行时证据、屏幕、报表、SME 判断，一路反推成可审计的 spec、traceability 和 handoff。**

这个边界一旦清楚，整条链路就顺了很多。

数据字典不再是“仓库里自己造一套主数据平台”。

RAG 不再是“替团队自动写答案”。

而是：

- 字典中心负责标准语义
- RAG 负责证据上下文
- Legacy Spec Factory 负责 legacy 事实反推和规格沉淀
- 下游 SDLC 负责把规格继续落成新系统实现

这其实是一个非常现实的设计决定。

因为企业里的现代化从来不是一个工具单点获胜，而是多个层次之间怎么分工。

如果边界不清楚，所有东西都会混在一起，最后谁都接不住。

## 二、第二个关键收敛：默认入口不再是“源码优先”，而是“模块优先”

这一周还有一个非常关键的变化，是我们把默认工作路径切到了 **module-first**。

这背后是现场经验，而不是理论偏好。

传统逆向分析最自然的路径，是从源码一路往上推：

`inventory -> program analysis -> flow analysis -> module synthesis -> BRD -> spec -> handoff`

这条路当然完整。

但真到企业现场，你会发现它有个很大的问题：

**太容易把项目拖成“考古工程”。**

源码量一大，程序一多，前半程就可能消耗掉大量时间和耐心。团队会陷在“还没看完”“还没理清”“还差一点”的状态里，迟迟进入不了真正面向交付的阶段。

而真实项目里，更常见的情况其实是：

- 业务方已经知道这次改的是哪个模块
- 老 SME 脑子里大致知道这块边界
- 团队已经能给出几条关键 flow
- 外部 RAG / code knowledge graph 也已经能查到大量上下文

在这种前提下，如果还坚持每次都从“全量源码挖掘”起步，很多工作其实是重复的。

所以这一周，我们把这条默认路径明确成了：

**RAG 辅助、模块优先。**

也就是：

`RAG evidence bundle + 四条 flow + SME notes -> module-context-intake -> module-analyzer -> brd-writer -> spec / handoff`

而源码级的 inventory、program、flow、data、screen 分析，不再是所有项目的唯一入口，而是作为 **verification path** 存在。

什么时候启用这条 verification path？

当模块边界不清楚的时候，
当 RAG 输出和人理解冲突的时候，
当某条规则风险很高需要补证的时候，
当 SME 觉得某个行为“不像真的”时，
再回头调 source-first 的技能去查证。

这一步的意义非常大。

它其实是在承认一个事实：

> **现代化项目不是学术逆向工程，真正重要的是尽快围绕“业务模块”形成可信规格，而不是形式上把所有源代码先遍历一遍。**

这和前一篇讲的“螺旋迭代”是高度一致的。所谓“螺旋迭代”，不是先把所有未知一次性清空，而是先用已有模块认知搭一个可工作的骨架，再在高风险点、冲突点和证据缺口上反复补证、回查、修正。

只是这一次，我们不只是在概念上说它，而是把它写进了默认工作流。

## 三、第三个关键点：RAG 可以给上下文，但不能替代真相

如果说这一周有什么原则是反复被确认的，我会选这一句：

**RAG 是证据上下文，不是最终真相。**

这句话非常重要。

因为只要开始让 AI 参与 legacy 分析，几乎一定会遇到一个问题：

它很擅长总结，也很擅长补全。

但 legacy 现代化最怕的，恰恰就是“补全”。

名字像一个业务概念，它就顺手把它写成业务规则；
代码里出现了一个字段，它就猜这个字段的业务口径；
RAG 返回了一段上下文，它就把这段上下文提升成事实。

这种问题最危险的地方在于，它写出来往往很像对的。

所以这一周，我们专门把“evidence-first anti-hallucination protocol”明确下来。

核心不是一句泛泛的“要谨慎”。

而是一套真正可以执行的规则：

- 没有 evidence 的内容，不能写成事实
- inferred business rule 不能直接进入正式 spec
- unknown / TBD 不能假装成 requirement
- modernization decision 不能写成 legacy behavior
- 矛盾不能被 AI 默默抹平，必须显式记录
- downstream 是否允许继续传递，必须标清楚

换句话说，我们开始给 AI 定一套“发言资格规则”。

它可以帮助整理。
它可以帮助归纳。
它可以帮助发现模式。
它甚至可以先给出建议。

但它不能越过证据和评审门禁，直接把猜测变成制度。

这背后其实是一个很现实的判断：

**在 legacy 现代化场景下，AI 最大的价值，不是替人直接得出结论，而是把事实、推断、冲突和待确认项组织得足够清楚。**

只有这样，后面 SME 和交付团队才能真正接得住。

## 四、第四个变化：我们不再把 SME review 当成一次会议，而是把它设计成一种默认交互

这一周我自己感受最强的一个变化，是我们对 SME 参与方式的改造。

以前大家说“SME 很重要”，通常怎么做？

先写一版材料，
再约个会，
再过一遍文档，
再现场记录一些结论，
然后会后再整理成纪要。

这当然不是不能用。

但它不适合高频迭代，也不适合 AI 协同。

因为 AI 擅长的是：

- 批量整理问题
- 风险排序
- 状态追踪
- 结构化回写

它最不擅长的，是一次性会议里的碎片化口头信息。

所以这一周，我们把 **Conversation Review Mode** 定成了默认模式。

这意味着什么？

意味着 SME review 不再是流程图最后一个框，而是整个系统里的持续决策机制。

AI 先从 draft BRD package 里，把 review items 整理出来。

然后按风险顺序，一次只抛 3-7 个问题给 SME：

- 先处理 blocking TBD
- 再处理 contradictions
- 再处理低置信度 BR
- 再处理高风险规则
- 再处理边界和异常场景
- 最后才是普通行为确认

每个问题都带着：

- target ID
- 一个足够业务化的问题表述
- evidence IDs
- confidence
- AI suggestion
- 可以直接回复的简短选项

这样 SME 不需要先完整读完几十页材料，才能开始有价值地参与。

他只需要围绕明确的问题做判断。

这意味着 SME 的一次参与，可以很轻量地长这样：

比如：

`BR-001 confirm`
`BR-003 reject, this is only a workaround`
`TBD-002 non-blocking`

然后这些回答会被结构化地写回工件里，而不是继续散落在聊天记录或会议纪要里。

这个变化的意义，我觉得很大。

因为过去大家总觉得 SME 稀缺，是因为 SME 太忙。

但很多时候，更真实的原因是：

**我们过去要求 SME 用一种最重、最慢、最不适合专家参与的方式协作。**

如果 AI 已经把问题收敛好了，把证据挂好了，把影响面标好了，那 SME 真正要做的，往往只是决策，而不是陪团队一起重新梳理背景。

这才是更像生产方式的人机协作。

## 五、第五个关键判断：我们做的不是“分析文档”，而是一层中间规格

这一周还有一个越来越明确的共识，是对最终输出物的重新定义。

很多传统项目里，“需求梳理”的终点是一份文档。

文档写完，评审通过，交给下游团队，然后后面的设计、开发、测试各自再翻译一遍。

这套模式的问题在于：

它对人类评审还算友好，
但对自动化和 AI 接力极不友好。

因为 prose 可以读，但不容易被继续消费。

所以 `legacy-spec-factory` 从一开始就把 `spec.yaml` / `spec.md` 作为核心结果，这周我们也越来越确定，这个方向是对的。

`spec.md` 是人读的。

`spec.yaml` 是机器读的。

前者解决沟通和评审问题，
后者解决自动化接力和下游消费问题。

再往下，还有 validation scenarios、traceability package、golden master test plan、BRD-to-SDD handoff。

这些东西连起来，构成的不是一堆“分析材料”，而是一层非常关键的 **specification layer**。

上游是 legacy 世界：

- RPG / CL / DDS
- job log / spool
- screen / report
- SME 记忆
- dictionary center
- RAG evidence bundle

下游是新系统世界：

- Java services
- APIs
- tests
- migration scripts
- deployment artifacts

中间如果没有一层结构化、证据化、可审计、可持续消费的规格层，现代化项目就会一直卡在两个低效状态之间：

要么开发团队被迫直接读老代码；
要么大家围着一堆半结构化文档反复解释。

所以这一周最本质的进展，不是“我们又写了几个 skill”。

而是我们越来越像在构建一层真正的中间协议。

这层协议，才是 legacy 世界和 AI-native SDLC 之间真正能长期复用的桥。

## 六、从提交表面看是功能增长，从本质上看是链路闭环

如果只看提交记录，这一周当然会显得像是“又加了不少能力”。

比如我们补了模块优先入口，补了 RAG 证据包的接入方式，补了聊天式 SME review，也补了运行时证据和对外预览能力。这里面有些词如果只看名字会很工程化，比如 `runtime evidence miner`，本质上是在把 job log、spool 这类运行现场证据结构化，让“系统真的这么跑过”这件事能被后续分析直接消费；`HTML exporter` 看起来像小工具，但它解决的是“规格和评审材料怎么更顺滑地发给 SME 或业务方预览，而不强依赖某种内部环境”。

但如果只把这些理解成“功能增量”，其实会低估这一周真正发生的变化。

真正的变化是：

**一条从输入到输出的链，开始闭环了。**

### 本周形成的链路全貌

`外部检索与代码知识图谱 -> 模块上下文 -> 模块分析 -> draft BRD package -> chat-driven SME review -> final BRD 与 spec -> traceability -> handoff -> downstream SDLC`

这意味着什么？

意味着我们不再只是“能分析一些东西”。

而是开始具备一种真正可执行的交付机制。

过去很多 AI 现代化项目，最容易失败的地方，就是卡在中间层：

前面分析很热闹，
后面没人接；

前面 insight 很多，
后面没有 contract；

前面输出很丰富，
后面无法复用。

只要链路闭不上，团队最后还是会退回老办法：

开更多会，
写更多解释材料，
让更多人脑二次翻译。

而这一周，我觉得最值得复盘的，恰恰是这条链路开始闭环。

它说明我们已经不是在验证一个概念，
而是在验证一种生产方式。

## 七、如果要把这一周和前一篇方法论放在一起看，我会这样理解

前一篇讲的是：

**为什么老系统需求必须反推，而不能只靠访谈和文档采集。**

这是方法论层面的突破。

这一周我们推进的是：

**当“反推”成为共识之后，怎么把它变成一条工程链路。**

这条链路里，最重要的不是某个单点技术，而是几个关键设计一起成立：

- 字典中心和反推系统边界清楚
- 默认入口从源码优先切到模块优先
- RAG 被定义为证据上下文，而不是答案引擎
- AI 的推断被纳入证据协议和门禁
- SME review 从一次性会议变成持续交互
- 输出物从分析文档升级成结构化规格层
- 最终结果可以继续喂给下游 SDLC，而不是停在人工解释阶段

如果说 Dogo 的文章是在回答“这条路为什么值得走”，

那这一周 `legacy-spec-factory` 的探索，回答的就是：

> **这条路如果真要走通，团队到底该怎么铺。**

这也是我越来越相信这件事的原因。

它不是一个漂亮的 AI demo。
不是一个逆向分析器集合。
也不是一个文档自动化项目。

它更像是一个正在成形的基础设施：

把遗留系统里的行为事实、业务语义、专家判断和现代化交付，接成一条连续的链。

这条链一旦站住，遗留系统现代化的起点就会发生变化。

它不再高度依赖“某个最懂系统的人今天还在不在”。

也不再每次改造都从头考古一次。

它会慢慢变成一种可以复制、可以协作、可以放大的生产能力。

而 `legacy-spec-factory` 这一周最值得复盘的地方，就在这里。

**我们不只是在理解 legacy。**

**我们是在尝试把“理解 legacy”这件事，变成未来软件工厂里一条可以稳定运行的产线。**
