# Golden Master Test Case Selection Protocol

This document formalizes how to choose which cases to include in a golden master
test plan.

## Selection Criteria

A test case (`TC-*`) should be included when:

1. **It validates an approved business rule or acceptance criterion.**
   - Every business-critical rule must have at least one test case or a named
     SME deferral.
   - Non-critical rules may be deferred with SME approval.

2. **It covers a distinct business scenario.**
   - Happy path (success case with normal data)
   - Rejection path (business logic refuses the transaction)
   - Exception path (system error, missing data, timeout)
   - Boundary / edge case (minimum value, maximum value, null, empty, duplicate)
   - Side effects (report output, batch behavior, DB state change)

3. **It is grounded in redacted, approved runtime evidence.**
   - Input samples must exist and be redacted.
   - Expected outputs must be observed from the legacy system, not inferred
     from the spec.

## Coverage Strategy

### Must Test

- Every business-critical approved `BR-*`
- Every approved `AC-*` marked as "golden_master_eligible"
- Happy path (at least one successful transaction per major flow)
- Explicit rejection paths named in the spec (e.g., "customer credit limit exceeded")
- Data validation errors (missing required fields, invalid format)

### Should Test

- Boundary conditions (minimum amount, maximum dates, boundary values)
- Idempotency (same input applied twice yields same result)
- Ordering (list output order matches legacy behavior)
- Concurrent / overlapping transactions (if the spec describes this)
- Report and batch-output side effects

### May Defer

- Non-critical secondary rules (with SME approval)
- Rare exception paths (if no runtime evidence and SME agrees)
- Performance characteristics (not golden master scope)
- UI styling or non-functional properties

### Do NOT Test

- Unrelated capabilities (separate test plan)
- Target-system implementation details (not golden master scope)
- Code generated by this test plan
- Hypothetical scenarios without evidence

## Avoiding Redundant Cases

**Do not include two test cases that validate the same rule with the same evidence.**

Example:

```
WRONG:
  TC-001: Valid order with positive amount -> expects success
  TC-002: Valid order with positive amount -> expects success

CORRECT:
  TC-001: Valid order with positive amount -> expects success
  TC-002: Valid order with zero amount -> expects rejection (boundary)
  TC-003: Order exceeding credit limit -> expects rejection (business rule)
```

## Handling Missing Evidence

If runtime evidence does not exist for a required case:

1. Mark it as a **blocking finding** (`FIND-*`).
2. Name the resolver (evidence team, SME, test data owner).
3. **Do not create placeholder values or invent expected outputs.**
4. The test plan remains `draft` or `blocked` until evidence is available or SME
   approves a deferral.

## SME Approval of Case Selection

Before marking the plan `approved`, the capability-owner SME must approve:

- The set of selected cases
- The business scenario each case represents
- The expected outputs (are they truly observed, not inferred?)
- Any deferrals (with reason)

The approval record must include:

- SME name and role
- Date of approval (ISO format)
- List of `TC-*` IDs approved
- Any caveats or conditions

## Relationship to Acceptance Criteria

Each `AC-*` may be covered by:

- **Exactly one `TC-*`** if the AC is narrow and self-contained
- **Multiple `TC-*`** if the AC spans several scenarios (happy path, multiple
  rejection paths, edge cases)
- **Zero `TC-*`** if the AC is deferred (marked in the test plan with reason and
  SME approval)

Avoid:

- Vague acceptance criteria that don't clearly map to test cases
- Acceptance criteria that combine unrelated scenarios (split them first in the
  spec)

## Coverage Summary

The `equivalence-coverage.md` matrix must show:

- Every business-critical `BR-*` with at least one `TC-*` or a marked deferral
- Every approved `AC-*` with at least one `TC-*` or a marked deferral
- Every `TC-*` linked to at least one `BR-*` or `AC-*`
- No orphaned test cases or business rules

## Revision

If the spec changes or new evidence appears:

1. **Do not renumber existing `TC-*` IDs.** Preserve ID stability.
2. **Add new `TC-*` cases** if new rules or scenarios emerge.
3. **Mark retired `TC-*`** if a case becomes obsolete (status: retired).
4. **Update `equivalence-coverage.md`** to reflect the new state.
5. **Route to SME for re-approval** if additions affect business-critical
   coverage.
